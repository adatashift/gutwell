## The Problem I Was Solving

A real person needed a simple, personalised tool to:

- Track daily gut health symptoms
- Follow a structured 3-phase healing protocol
- Log meals, herbs, and daily habits
- Keep a private journal with progress history
- Export data to share analyse and possibly share with a health practicioner 

So I built one.

---

## How I Built It

### 1. I defined the problem before touching anything

Before asking AI to build anything, I mapped out:
- What does this person actually need?
- What are the must-have features?
- What does success look like?

Clear thinking at the start saved enormous time later.

### 2. I translated the problem into features

I broke the solution into manageable parts:
daily check-in, phase protocol, meal guidance, herb rotation, 
journal, data storage, and progress tracking.

Each piece was scoped before it was built.

### 3. I used AI to generate, then I evaluated

I did not accept the first output. Every version was tested, 
challenged, and improved. The AI was a capable collaborator — 
but the judgment was always mine.

### 4. I debugged by thinking, not guessing

When something broke I isolated the problem, asked targeted 
questions, and fixed one thing at a time. Trying to fix 
everything at once never works.

### 5. I manually edited the code on GitHub

Rather than always replacing the entire file, I learned to 
make targeted edits directly in the GitHub editor — finding 
specific lines, understanding what each block of code does, 
and changing only what needed to change.

This gave me a working sense of how the code is structured:
- Where the visual styling lives (CSS)
- Where the page content lives (HTML)
- Where the logic and behaviour lives (JavaScript)

It is not the same as being a developer. But it is no longer 
a black box either.

### 6. I shipped when it was good enough

Knowing when to stop is its own skill. Once the app was 
stable, usable, and solving the real problem — I stopped 
adding and started refining.

### 7. I made responsible decisions about what to include

As the project developed I made deliberate choices about 
features — including removing the live AI chat assistant 
to ensure the app remained safe (in the event that someone chose to use the app for daily tracking), predictable, and appropriate 
for general use. Knowing what to leave out is as important 
as knowing what to build.

## What I Learned

**Problem-solving matters more than technical knowledge.**
You do not need to know everything. You need to know how to 
break things down, ask the right questions, and think clearly.

**AI is a tool, not a shortcut.**
Good results came from clear instructions, the right context, 
and iteration — not from asking once and hoping for the best.

**Precision changes everything.**
The more specific I was, the better the output, the fewer 
the errors, and the faster the progress.

**Simplicity is a decision, not a limitation.**
A working, usable solution is worth more than a complex, 
incomplete one. I made that choice deliberately.

**Knowing what to remove matters as much as knowing what to build.**
Removing the AI chat was a deliberate responsible decision — 
not a limitation. Understanding risk, liability, and user 
safety is part of building anything real.

**Iteration is the whole game.**
The first version was not good. That was expected and fine. 
Every meaningful improvement came from testing, adjusting, 
and repeating.
